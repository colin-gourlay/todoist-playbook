# Repo Profile Audit

A baseline audit to align public references to a repository — establishing a canonical identity, verifying consistency across social and professional profiles, and tracking any references that cannot be resolved immediately.

This template is derived from the [Socials Health & Optimisation Checklist](../socials-health-and-optimization-checklist/) and narrows its scope to a single repository. Use it whenever a repository's public presence feels inconsistent, after a rename or ownership transfer, or as a one-time discoverability investment.

---

## Objective

- Inventory every public-facing location where the repository is referenced
- Define a canonical name, one-line description, short blurb, and URL for the repository
- Update all controllable references to use the canonical identity
- Validate that all links and URLs are active and correctly targeted
- Document any unresolved references as tracked follow-up items

---

## When to Use

- After creating or launching a new repository
- After renaming a repository or transferring ownership
- When references across profiles feel inconsistent or outdated
- Before a major promotion effort (Awesome List submission, blog post, social campaign)
- As a periodic hygiene check — recommended every 6 months for actively maintained repos

---

## Structure Overview

1. **Baseline Audit — Inventory All References** — surface every public location where the repo is mentioned
2. **Define Canonical Identity** — lock in the definitive name, description, blurb, URL, and topics
3. **GitHub Repository Hygiene** — bring the repository's own About panel, README, and social preview into line
4. **Social & Professional Profile Updates** — update LinkedIn, GitHub profile, X, personal site, and other profiles
5. **Directory & Listing Updates** — update package registries, Awesome Lists, and developer directories
6. **Link & URL Validation** — confirm every link resolves correctly and no stale URLs remain
7. **Unresolved References & Follow-Up Tracking** — log anything that could not be fixed, with reasons

---

## Canonical Identity Reference

Before working through sections 3–6, complete section 2 and save the following in a single document you can paste from as you update each platform:

| Field | Value |
|-------|-------|
| Canonical name | _(e.g. Todoist Playbook)_ |
| One-line description | _(≤ 160 chars, describes what + why)_ |
| Short blurb | _(2–3 sentences for bios and directories)_ |
| Canonical URL | _(e.g. https://github.com/owner/repo)_ |
| Topics / tags | _(comma-separated keywords)_ |

Having this reference in one place prevents copy-paste drift across platforms.

---

## What to Fix vs. What to Track

Not every reference can be updated. Apply the following rule:

- **Fix it now** — any profile, README, or directory you control directly
- **Track it** — any external article, archived page, or third-party listing you cannot edit
- **Create a follow-up task** — any reference that can be fixed but requires significant effort (e.g. opening a PR, contacting a maintainer)

Document tracked items in section 7 so there is a clear record of known gaps.
