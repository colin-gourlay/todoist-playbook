# Sprint Review

A structured checklist for running a collaborative sprint review — designed to move beyond demo-only sessions toward meaningful stakeholder engagement, real-time backlog adaptation, and clear alignment on product direction.

---

## Objective

- Establish a consistent, repeatable format for all Sprint Reviews
- Demonstrate delivered value through user journeys, not ticket walkthroughs
- Actively capture stakeholder feedback and convert it into backlog items
- Improve transparency and maintain alignment on product direction

Estimated duration: 60–90 minutes (including preparation and follow-up).

---

## When to Use

- At the end of every sprint or iteration
- As a recurring ceremony within a broader Sprint Ceremonies Kit
- Whenever stakeholder alignment or backlog prioritisation needs a reset

---

## Structure Overview

1. Before the Review — preparation, invites, and demo readiness
2. Welcome & Context — restate the Sprint Goal and set expectations
3. Increment Demo — demonstrate completed work via user journeys
4. Stakeholder Feedback — open discussion and real-time feedback capture
5. Product Backlog Review — adjust priorities and refine items live
6. Metrics & Progress — optional velocity and release progress summary
7. Wrap-Up — confirm next steps and key decisions
8. After the Review — convert feedback, update backlog, share notes

---

## Definition of Done

Only work that meets the Definition of Done should be demonstrated. This typically means:

- Feature is complete and tested
- Code is reviewed and merged
- Acceptance criteria are satisfied
- No known blocking defects

---

## Suggested Ritual

- Prepare demo scenarios in advance — focus on user journeys, not implementation details
- Ensure stakeholders are invited and have context before the session
- Capture feedback in real time — do not defer to memory
- Product Owner should update the backlog during the session where possible
- Do not skip the "After the Review" section — unlogged feedback is lost feedback

---

## Anti-Patterns to Avoid

- Treating the review as a one-way demo with no stakeholder interaction
- Showing incomplete or partially done work
- Walking through tickets rather than user journeys or business outcomes
- Leaving the session without backlog updates
- Turning the review into a status report or team retrospective

---

## Import Instructions

1. Download `template.csv`
2. Create a new project in Todoist
3. Import from CSV
4. Rename the project to: `Sprint Review – Sprint [N]`
5. Repeat for each sprint (typically every 1–2 weeks depending on your sprint length)
